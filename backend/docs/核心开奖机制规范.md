# 核心开奖机制规范

> 版本：2.0（已实施）
> 最后更新：2026-05-13
> 适用范围：香港彩 (ID=1)、澳门彩 (ID=2)、台湾彩 (ID=3)
> 实现文件：`backend/src/crawler/scheduler.py`、`backend/src/crawler/result_crawler.py`

## 一、开奖总流程概览

### 1.1 时序流程

```
时间轴（北京时间）                    动作                                   负责函数
─────────────────────────────────────────────────────────────────────────────────────────
T - 24h                              每日预测任务（对所有活跃彩种）              daily_prediction 任务
T - 300s                             动态轮询间隔缩短至 10s                     _compute_dynamic_crawl_interval()
T - 60s                              调度器确认 next_time 正确                  _reschedule_precise_checks()
T - 1s                               精确定时器触发                             threading.Timer
T + 0s                               开奖时间到达（外部 API 公布结果）           外部数据源
    ├── Phase 1                      验证期号匹配（最多 4 次重试）               _do_precise_draw_fetch_and_open()
    ├── Phase 2   ↓ 匹配             拉取完整开奖记录（numbers/open_time/next）  _fetch_current_draw_records()
    ├── Phase 3   ↓ 成功             Upsert 到 lottery_draws                    _upsert_current_draw_records()
    ├── Phase 4   ↓ 成功             立即开盘 is_opened=1（延迟 ≤ 1s）           _open_specific_records()
    ├── Phase 5   ↓ 成功             延迟回填开奖结果                            _schedule_backfill_after_draw()
    └── Phase 6                       同步 next_time → system_config            sync_all_lottery_type_next_times()
T + 2~15s                            前端 API 可查询到最新开奖结果               GET /api/*/draws
    ↓ 全部重试失败                    触发告警 + 切换追赶模式（5s 加速轮询）       _check_staged_timeout_alerts()

    预测生成（独立于开奖管线）：
T + 次日 12:00                       daily_prediction 任务统一为所有彩种生成下一期预测
```

### 1.2 HK/Macau vs Taiwan 差异化路径

| 阶段 | 香港彩 (1) / 澳门彩 (2) | 台湾彩 (3) |
|------|------------------------|-----------|
| 数据来源 | 外部 API `lnlllt.com/api.php` 自动抓取 | 管理后台手工 JSON 导入 |
| 触发方式 | 精确 Timer（`next_time - 1s`）→ 完整抓取管线 | 精确 Timer → `_open_taiwan_draws_and_update_next_time()` |
| 期号验证 | 4 次重试 + 期号不匹配告警 | 无（手工录入） |
| 开盘 | upsert 后立即 `_open_specific_records()` | 从 `lottery_draws` 标记 `is_opened=1` |
| 回填 | `_schedule_backfill_after_draw()`（延迟回填开奖结果） | 同左 |
| 预测生成 | `daily_prediction` 每天 12:00 统一批量生成 | 同左 |

**统一设计原则**：开奖管线只负责数据抓取 + 入库 + 开盘 + 回填。预测生成由独立的
`daily_prediction` 定时任务（每天 12:00 北京时间）对所有彩种统一处理。这样：
- 开奖管线不依赖预测生成的成功与否，更简洁、更可靠
- 预测生成逻辑集中在 `_run_auto_prediction`，方便维护和调试
- 避免开奖时立即生成预测可能与 daily_prediction 产生竞态

### 1.3 关键节点 SLA

| 节点编号 | 节点名称 | 触发源 | 最大允许耗时 | 超时动作 |
|---------|---------|--------|------------|---------|
| N1 | 精确检查触发 | `threading.Timer` 在 `next_time - 1s` | ≤ ±1s | — |
| N2 | 期号验证（含重试） | N1 完成后 | ≤ 10s（4 次 × 2s 间隔） | 全部失败 → mismatch 告警 + 追赶模式 |
| N3 | 完整数据拉取 | N2 期号匹配后 | ≤ 10s（HTTP 超时 30s） | 失败 → 告警 + 追赶模式 |
| N4 | 数据 upsert | N3 返回后 | ≤ 2s | 失败 → 日志 + 下一轮重试 |
| N5 | 自动开盘 | N4 成功后 | ≤ 1s | 失败 → 日志记录 |
| N6 | 延迟回填 | N5 完成后 | 默认 5 分钟延迟 | 回填任务入队失败 → 日志 |
| N7 | 前端可查询 | N5 完成后（无需等 N6） | ≤ 200ms | 超过 → `slow_call_warning` |

**端到端 SLA（从 T+0s 到 N5 完成）：≤ 15 秒**

> **注**：预测生成（原 N6）已从开奖管线中移除。改为每天 12:00 由 `daily_prediction` 任务统一处理，
> 该任务遍历所有活跃彩种，回填最新开奖结果并生成下一期预测。

## 二、各彩种开奖时效性标准

### 2.1 时间精度要求

| 指标 | 标准值 | 实现方式 |
|------|--------|---------|
| 精确定时器触发精度 | ≤ ±1s（相对于 `next_time`） | `threading.Timer(delay, _fire)` |
| `next_time` 存储精度 | 毫秒级 Unix 时间戳（UTC） | `_calc_and_update_next_time()` 写入 |
| `draw_time` 存储格式 | 北京时间字符串 `YYYY-MM-DD HH:mm:ss` | `lottery_draws.draw_time` 列 |
| next_time 空缺回退 | 使用默认开奖时间计算 | `_compute_hk_macau_default_next_time_ms()` |

### 2.2 各彩种开奖时间配置

| 彩种 | 默认开奖时间（北京时间） | 配置键 | 主采集 URL | 备用采集 URL |
|------|----------------------|--------|-----------|------------|
| 香港彩 | 21:30 | `draw.hk_default_draw_time` | `draw.hk_default_collect_url` | `draw.hk_backup_collect_url` |
| 澳门彩 | 21:30 | `draw.macau_default_draw_time` | `draw.macau_default_collect_url` | `draw.macau_backup_collect_url` |
| 台湾彩 | 22:30 | `draw.taiwan_default_draw_time` | N/A（手工 JSON 导入） | N/A |

### 2.3 数据时效硬性指标

| 指标 | 硬性上限 | 测量起点 | 分级告警 |
|------|---------|---------|---------|
| 数据抓取延迟 | ≤ 15s | 开奖时间 (T+0) | 黄色 30s / 橙色 120s / 红色 300s |
| 自动开盘延迟 | ≤ 1s | upsert 完成 | > 15s 日志 WARNING |
| 前端展示延迟 | ≤ 200ms | API 请求到达 | > 500ms 慢查询告警 |

## 三、准时保障机制

### 3.1 多重时间校准

| 层级 | 机制 | 函数 | 容错 |
|------|------|------|------|
| 配置层 | `lottery.{type}_next_time` 由最新开奖记录自动推导 | `sync_lottery_type_next_time_from_latest_draw()` | 为空时回退到 `_compute_hk_macau_default_next_time_ms()` |
| 调度层 | `sync_all_lottery_type_next_times()` 多源同步 | 每次爬取/开盘/启动后调用 | 同步失败记录日志 |
| 校验层 | `_do_precise_draw_fetch_and_open()` 期号比对 | 开奖前 1s 触发 | 不匹配重试 4 次，全失败则告警 |

### 3.2 自动化开奖程序

#### 3.2.1 香港彩 / 澳门彩 —— 精确触发 + 完整抓取管线

实现函数：`CrawlerScheduler._do_precise_draw_fetch_and_open()`

```
触发源：threading.Timer，fire_at = next_time - 1s
触发动作（6 个 Phase 顺序执行）：
  Phase 1 — 验证期号：
    1. 从 system_config 读取 current_period，计算 expected_period
    2. 调用 _fetch_current_draw_period() 获取外部 API 当前期号
    3. 如果期号不匹配 → 等待 2s 重试（最多共 4 次）
    4. 全部失败 → _trigger_draw_mismatch_alert() + 邮件告警 + 追赶模式
    5. 期间若 next_time 被其他路径更新 → 推迟检查
  Phase 2 — 拉取完整数据：
    1. 调用 _fetch_current_draw_records()（使用主/备 URL + HTTP 重试）
    2. 获得标准化 records 列表（issue, open_time, result, next_time）
    3. 无数据 → 失败审计 + 告警 + 追赶模式
  Phase 3 — 数据入库：
    1. _upsert_current_draw_records() 按 year+term 比对 upsert
    2. 记录审计日志（precise_upsert）
  Phase 4 — 立即开盘：
    1. _open_specific_records() 按 year+term 精确开盘 is_opened=1
    2. 延迟 ≤ 1 秒（无轮询等待）
    3. 记录审计日志（precise_open）
  Phase 5 — 延迟回填：
    1. _schedule_backfill_after_draw() 安排回填任务（默认 5 分钟后执行）
    2. 回填任务调用 _backfill_latest_opened_prediction_results() 更新 res_code/res_sx/res_color
    3. 预测生成由 daily_prediction 每天 12:00 统一处理，不在此阶段执行
  Phase 6 — 同步：
    1. sync_all_lottery_type_next_times() 更新 next_time
    2. 写入完整管线审计日志（precise_complete）
```

#### 3.2.2 台湾彩 —— 精确定时 + 开盘联动

实现函数：`CrawlerScheduler._open_taiwan_draws_and_update_next_time()`

```
触发源：threading.Timer，fire_at = next_time - 1s
触发动作：
  1. 将 draw_time <= now 的未开盘 type=3 记录置为 is_opened=1
  2. 计算并更新下一期 next_time（draw_time + 1 天 → UTC 毫秒时间戳）
  3. 延迟 5 分钟执行预测回填（_schedule_backfill_after_draw）
  4. 重新调度下一期精确检查
```

### 3.3 动态轮询加速

实现函数：`CrawlerScheduler._compute_dynamic_crawl_interval()`

`_auto_crawl` 作为精确检查的安全兜底，使用动态间隔：

| 模式 | 间隔 | 触发条件 |
|------|------|---------|
| 追赶模式 | 5s | `_chase_modes[lt_id] == True`（开奖超时未获数据） |
| 开奖窗口 | 10s | 当前时间在任一活跃彩种 `next_time ± 5min` 内 |
| 平时 | 300s | 不在任何开奖窗口内 |
| 最小值 | 5s | 系统硬限制 |
| 最大值 | 3600s | 系统硬限制 |

追赶模式由 `_set_lottery_chase_mode()` 控制，以下情况进入追赶：
- 精确检查期号不匹配（全部重试失败）
- 期号匹配但数据拉取返回空
- 分级告警检测到开奖超时（黄色/橙色/红色）

追赶模式在以下情况退出：
- 数据成功 upsert
- 分级告警确认数据已到达

### 3.4 冗余数据源与自动切换

实现函数：`_get_effective_collect_url()`

| 彩种 | 主 URL 配置键 | 备用 URL 配置键 | 切换条件 |
|------|------------|-------------|---------|
| 香港彩 | `draw.hk_default_collect_url` | `draw.hk_backup_collect_url` | 连续失败次数 ≥ `crawler.backup_fail_count_threshold`（默认 2） |
| 澳门彩 | `draw.macau_default_collect_url` | `draw.macau_backup_collect_url` | 连续失败次数 ≥ `crawler.backup_fail_count_threshold`（默认 2） |

切换逻辑：
- `_get_effective_collect_url()` 读取 `_crawler_fail_count_key(lt_id)` 的失败计数
- 达到阈值且备用 URL 非空时，返回备用 URL 作为主 URL
- 备用 URL 不可配置时，仅使用主 URL
- HTTP 层 (`fetch_current_term_data`) 还支持主 URL 全部失败后自动尝试备用 URL

### 3.5 实时监控告警规则

| 告警名称 | 触发条件 | 级别 | 函数 |
|---------|---------|------|------|
| 期号不匹配告警 | 精确检查 4 次期号不匹配 | ERROR | `_trigger_draw_mismatch_alert()` + `alert_precise_draw_mismatch()` |
| 黄色超时预警 | 开奖后 30s 数据未入库 | WARNING | `_check_staged_timeout_alerts()` |
| 橙色超时告警 | 开奖后 120s 数据未入库 | ERROR | `_check_staged_timeout_alerts()` |
| 红色超时告警 | 开奖后 300s 数据未入库 | CRITICAL | `_check_staged_timeout_alerts()` |
| 爬虫失败告警 | 连续失败次数 ≥ `alert.crawler_retry_threshold` | ERROR | `alert_crawler_failure()` |
| 数据停滞告警 | 最新开奖记录 > 2 个开奖周期 | WARNING | `alert_draw_staleness()` |

## 四、迅速响应机制

### 4.1 结果生成后的即时分发

```
开奖数据 upsert 成功
    │
    ├── 立即开盘 _open_specific_records()              延迟 ≤ 1s
    │       └── is_opened=1 → 前端 API 立即可返回
    │
    ├── 延迟回填 _schedule_backfill_after_draw()       默认 5 分钟后
    │       └── created.* 表 res_code/res_sx/res_color 更新
    │
    ├── 同步 system_config（next_time / current_period）
    │       └── 为下一期精确检查做准备
    │
    ├── 写入 draw_audit_log（precise_complete）        审计可追溯
    │
    └── 预测生成 → 每天 12:00 daily_prediction 统一处理
```

### 4.2 前端展示延迟要求

| 接口 | 路径 | 目标延迟 | 最大允许 |
|------|------|---------|---------|
| 最新开奖结果 | `/api/draws/latest?type=N` | ≤ 50ms | ≤ 200ms |
| 历史开奖列表 | `/api/draws?type=N&page=1` | ≤ 100ms | ≤ 500ms |
| 预测资料 | `/api/predictions?type=N` | ≤ 100ms | ≤ 500ms |

## 五、延迟容错与恢复策略

### 5.1 异常切换至备用方案

| 异常类型 | 检测方式 | 最大决策时间 | 切换动作 |
|---------|---------|------------|---------|
| 主采集 URL 不可达 | HTTP 超时/错误/连接重置 | 10s（2 次 × 30s 超时） | `fetch_current_term_data` 自动尝试备用 URL |
| 备用 URL 也不可达 | HTTP 超时/错误 | 10s | 触发失败计数 + 告警 |
| 失败次数达阈值 | 失败计数 ≥ `backup_fail_count_threshold` | 立即 | `_get_effective_collect_url()` 返回备用为主 |
| 数据库写入失败 | DB 异常 | 立即 | 日志记录 + 告警 + 追赶模式 |
| next_time 为空 | 启动时/同步时检测 | 立即 | `_compute_hk_macau_default_next_time_ms()` 回退 |

### 5.2 人工兜底流程

| 条件 | 动作 |
|------|------|
| 开奖后 300s（红色告警）数据仍未入库 | 系统发送"需人工介入"告警邮件 |
| 管理员收到告警 | 登录后台 → 点击"更新彩种数据" → 调用 `run_crawl_only()` |
| 手动抓取也失败 | 管理员通过"手动录入开奖"功能录入号码 |

### 5.3 完整错误处理链路

```
外部 API 请求失败
    │
    ├── HTTP 层重试（retry_count 次，间隔 retry_delay 秒）
    │       └── result_crawler.py: fetch_current_term_data()
    │
    ├── 主 URL 全部失败 → 自动尝试备用 URL
    │       └── result_crawler.py: urls list iteration
    │
    ├── 全部 URL 失败 → increment_crawler_fail_count
    │       └── 达到阈值 → _get_effective_collect_url() 永久切换
    │
    ├── 精确检查管线失败 → _set_lottery_chase_mode(True)
    │       └── _compute_dynamic_crawl_interval() → 5s 加速追数据
    │
    └── 分级超时告警（30s/120s/300s）
            └── _check_staged_timeout_alerts() → 邮件 + 日志
```

## 六、合规与审计要求

### 6.1 开奖审计日志

表名：`draw_audit_log`（实现：`database/schema/audit.py`）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGSERIAL | 主键 |
| `lottery_type_id` | INT | 彩种 ID（1=香港, 2=澳门, 3=台湾） |
| `year` | INT | 年份 |
| `term` | INT | 期号 |
| `event` | TEXT | 事件类型，见下表 |
| `event_time` | TEXT | 事件时间（UTC ISO8601） |
| `duration_ms` | INT | 本事件耗时（毫秒） |
| `status` | TEXT | `success` / `failed` / `timeout` |
| `detail` | TEXT | 事件详情（URL、HTTP 状态码、记录数等） |
| `operator` | TEXT | 操作者：`scheduler` 或管理员用户名 |
| `created_at` | TEXT | 记录创建时间（UTC ISO8601） |

事件类型枚举：

| event 值 | 含义 | 写入位置 |
|----------|------|---------|
| `precise_fetch` | 精确检查数据拉取 | `_do_precise_draw_fetch_and_open()` Phase 2 |
| `precise_upsert` | 精确检查数据入库 | `_do_precise_draw_fetch_and_open()` Phase 3 |
| `precise_open` | 精确检查自动开盘 | `_do_precise_draw_fetch_and_open()` Phase 4 |
| `precise_complete` | 精确检查完整管线完成 | `_do_precise_draw_fetch_and_open()` Phase 6 |
| `auto_crawl_fetch` | 轮询爬取（兜底） | `_auto_crawl()` |
| `manual_entry` | 管理员手工录入 | 管理后台 API |
| `backfill` | 预测结果回填 | `_backfill_latest_opened_prediction_results()` |

写入函数：`_log_draw_audit()`（`scheduler.py` 模块级函数）

### 6.2 审计要求

1. **每次开奖必须有完整审计链路**：精确检查触发 → 数据拉取 → 入库 → 开盘 → 完成，每个 Phase 独立记录。
2. **审计日志保留不少于 90 天**。
3. **人工操作必须记录操作人用户名和操作时间**（`operator` 字段）。
4. **配置变更历史记录在 `system_config_history` 表，永久保留**。
5. **错误日志记录在 `error_logs` 表和 `data/error.log` 文件**。

### 6.3 审计查询接口

| 接口 | 用途 |
|------|------|
| `GET /api/admin/audit/draw?type=N&date=YYYY-MM-DD` | 查询指定日期开奖审计记录 |
| `GET /api/admin/audit/draw/term?type=N&year=Y&term=T` | 查询指定期号完整链路 |

## 八、预测生成入口锁定

### 8.1 授权入口（仅此两个）

| # | 入口 | 触发方式 | 类型 | 实现位置 |
|---|------|---------|------|---------|
| 1 | 管理后台手动触发 | 管理员点击"生成预测"按钮 | HTTP API | `POST /api/admin/lottery-types/{id}/crawl-and-generate`、`/api/admin/sites/{id}/prediction-modules/generate-all`、`/api/admin/sites/{id}/mode-payload/{table}/{row}/regenerate` |
| 2 | `daily_prediction` 定时任务 | 每天 `daily_prediction_cron_time`（默认 12:00 北京时间）自动触发 | `scheduler_tasks` | `_ensure_daily_prediction_task()` → `_execute_task()` → `_run_auto_prediction()` |

> **任何其他代码路径均不得触发预测数据生成。**

### 8.1.1 覆盖控制规则

为避免自动化任务误覆盖已有预测资料，系统区分“允许覆盖”和“禁止覆盖”两类入口：

| 类别 | 入口 | 是否允许覆盖既有预测内容 | 说明 |
|------|------|------------------------|------|
| 管理员手工入口 | `/api/admin/sites/{id}/prediction-modules/generate-all`、`/api/admin/sites/{id}/mode-payload/{table}/{row}/regenerate`、其他显式人工重生成路径 | 允许 | 管理员手工操作属于显式覆盖行为，底层仍通过 `upsert_created_prediction_row()` 更新同一期记录 |
| 自动化回填入口 | `backfill_after_draw` → `_backfill_latest_opened_prediction_results()` | 不覆盖预测内容 | 只允许回填 `res_code/res_sx/res_color`；且仅在这些字段为空时写入 |
| 自动化未来一期生成 | `daily_prediction` → `_run_auto_prediction()` 未来下一期分支 | 禁止覆盖 | 若目标未来期已存在预测内容，必须跳过，不得自动重生成 |
| 自动化近期期数补生成 | `daily_prediction` → `_ensure_recent_periods_have_predictions()` | 禁止覆盖 | 仅允许补“缺失的站点/模块/期数”，若该站点该期预测已存在，必须跳过 |

自动化任务当前的强制约束：

1. `daily_prediction` 只能补缺，不能重写已有预测正文。
2. 自动化链路可以回填开奖结果字段，但不能因此重算或覆盖已有预测内容。
3. 任何“重新生成预测正文”的行为，必须来自管理员显式手工操作。

### 8.2 已封堵的入口

| 原入口 | 处置 | 说明 |
|--------|------|------|
| `TASK_TYPE_AUTO_PREDICTION` | 处理器改为仅记录日志拒绝执行 | `_schedule_auto_prediction` 已注释，`_execute_task` 中遇到该类型直接 return |
| `_run_auto_prediction` 在开奖管线中 | 已移除 | 精确检查 Phase 5 和 `_auto_crawl` 改为调用 `_schedule_backfill_after_draw`（仅回填，不生成） |
| `enqueue_manual_daily_prediction_task` | 保留但无调用者 | 该函数创建 `daily_prediction` 类型的任务（属于授权入口 #2），但当前无任何代码调用 |

### 8.3 验证方法

在代码库中执行：
```
grep -rn "bulk_generate_site_prediction_data\|_run_auto_prediction\|bulk_generate_site_predictions" backend/src/
```

除 `daily_prediction` 任务路径和管理后台路由外，不得有其他调用。

自动覆盖保护应额外满足：

```bash
grep -rn "_ensure_recent_periods_have_predictions\|created_prediction_issue_exists\|upsert_created_prediction_row" backend/src/
```

检查点：

1. `_ensure_recent_periods_have_predictions()` 必须先做“是否已存在预测”的判断，再决定是否生成。
2. 自动化链路命中已存在预测时，应记录 `skipped_existing` 或等价语义日志，而不是进入重生成。
3. `upsert_created_prediction_row()` 允许保留通用 `UPDATE` 行为，但自动化补生成路径不得把已有记录送入该更新逻辑。

完整测试步骤参见 [daily_prediction 测试指南](./daily_prediction测试指南.md)。
