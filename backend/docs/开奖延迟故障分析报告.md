# 开奖延迟故障分析报告

> 适用彩种：香港彩 (ID=1)、澳门彩 (ID=2)、台湾彩 (ID=3)
> 分析日期：2026-05-13
> 涉及模块：`backend/src/crawler/scheduler.py`、`backend/src/runtime_config.py`、`backend/src/crawler/result_crawler.py`、`backend/src/helpers.py`

---

## 一、开奖链路完整节点

```
┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────┐
│ 外部 API  │ →  │  数据抓取     │ →  │  数据入库     │ →  │  自动开盘     │ →  │ 前端展示  │
│ 开奖结果  │    │  (爬虫/采集)  │    │ lottery_draws │    │ is_opened=1  │    │   API     │
└──────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────┘
       ↑               ↑                  ↑                   ↑                  ↑
   外部依赖         调度器触发           DB 写入             轮询检查          前端轮询/刷新
```

每个节点的现有机制和最大延迟：

| 节点 | 香港彩/澳门彩 | 台湾彩 | 当前最大延迟 |
|------|-------------|--------|------------|
| 数据抓取 | `_auto_crawl()` 每 600s 轮询 | 人工 JSON 导入 | **≤ 600s** |
| 数据入库 | `_upsert_current_draw_records()` | 手动导入后写入 | 取决于抓取频率 |
| 自动开盘 | `_auto_open_draws()` 每 60s | 精确定时器开奖 | **≤ 60s** |
| 前端 API | 查询 `lottery_draws WHERE is_opened=1` | 同左 | 取决于开盘 |

## 二、故障问题分类与根因分析

### 问题 1：精确检查不拉取数据（关键缺陷）

- **故障分类**：架构设计缺陷
- **影响程度**：★★★★★ 严重
- **影响彩种**：香港彩、澳门彩
- **故障现象**：开奖时间到达后，精确检查定时器在 `next_time - 1s` 触发，但仅做期号比对，不抓取实际开奖数据。数据须等待下一次 `_auto_crawl` 轮询（最坏 10 分钟）。
- **根因**：[scheduler.py:233-304](backend/src/crawler/scheduler.py#L233) `_do_precise_draw_check()` 的逻辑是：请求外部 API → 获取 `current_period` → 与本地 `expected_period` 比对 → 匹配则返回（**不做任何数据拉取**）。期号匹配成功后直接 `return`，没有触发 `_fetch_current_draw_records`。
- **代码位置**：[scheduler.py:273-278](backend/src/crawler/scheduler.py#L273)

```python
# 当前逻辑（有缺陷）：
if period and period == expected_period:
    _crawler_logger.info("Precise check lt=%s: period matched ...")
    return  # ← 仅仅返回，不拉数据！
```

- **改进方案**：期号匹配成功后，立即调用 `_fetch_current_draw_records()` + `_upsert_current_draw_records()`，并在插入新数据后立即执行 `_auto_open_draws()` 逻辑开盘。
- **预防措施**：将精确检查和数据抓取合并为一个原子操作，避免"验证通过但不拉数据"的半吊子流程。

---

### 问题 2：自动抓取轮询间隔过大

- **故障分类**：配置参数不合理
- **影响程度**：★★★★ 严重
- **影响彩种**：香港彩、澳门彩
- **故障现象**：`crawler.auto_crawl_interval_seconds` 默认 600 秒（10 分钟）。即使问题 1 修复，若精确检查调度失败，轮询也要等 10 分钟。
- **根因**：[scheduler.py:439-440](backend/src/crawler/scheduler.py#L439) 最小允许 30 秒，但默认值 600 秒。
- **改进方案**：将默认值从 600s 降低到 30-60s，在开奖时间窗口（±5 分钟）内加速到 10-15s。开奖窗口外恢复长间隔以节省资源。

---

### 问题 3：自动开盘轮询间隔延迟

- **故障分类**：配置参数可优化
- **影响程度**：★★★ 中等
- **影响彩种**：全部
- **故障现象**：`crawler.auto_open_interval_seconds` 默认 60 秒，数据入库后 `is_opened=1` 最长延迟 60 秒。
- **根因**：[scheduler.py:436-437](backend/src/crawler/scheduler.py#L436) 最小允许 5 秒，但默认 60 秒。且开盘逻辑和数据抓取是解耦的两个独立循环，没有触发式联动。
- **改进方案**：`_auto_crawl` 成功 upsert 后，立即对这批记录执行开盘，不需要等下一轮 `_auto_open_draws`。

---

### 问题 4：next_time 为空导致精确检查静默跳过

- **故障分类**：状态同步缺陷
- **影响程度**：★★★★ 严重
- **影响彩种**：香港彩、澳门彩
- **故障现象**：`lottery.hk_next_time` 或 `lottery.macau_next_time` 为空时，`_reschedule_precise_checks()` 直接 `continue` 跳过，不安排任何定时器。管理员可能完全不知道精确检查未运行。
- **根因**：[scheduler.py:481-489](backend/src/crawler/scheduler.py#L481)
- **改进方案**：
  1. next_time 为空时使用默认开奖时间（`draw.hk_default_draw_time`、`draw.macau_default_draw_time`）计算 next_time，而不是静默跳过。
  2. 启动时 next_time 同步失败应触发告警。

---

### 问题 5：调度器寄生在 HTTP 服务器进程内

- **故障分类**：架构风险
- **影响程度**：★★★★★ 严重
- **影响彩种**：全部
- **故障现象**：HTTP 服务器进程崩溃、重启、或未启动时，所有调度器停止工作。没有独立的守护进程或 watchdog。
- **根因**：[server.py:212-213](backend/src/app_http/server.py#L212) 调度器在 `run_server()` 中创建和启动。
- **改进方案**：
  1. 添加调度器心跳检测（写 `system_config` 表的时间戳），由外部 watchdog 或管理后台监控。
  2. 长期方案：将调度器拆分为独立进程，通过 `system_config` 和数据库任务表通信。

---

### 问题 6：外部 API 单点故障无冗余

- **故障分类**：外部依赖风险
- **影响程度**：★★★★ 严重
- **影响彩种**：全部
- **故障现象**：所有彩种的实时数据都依赖 `https://www.lnlllt.com/api.php`。API 不可达时，自动抓取直接失败，无备用数据源切换。
- **根因**：[scheduler.py:314-316](backend/src/crawler/scheduler.py#L314) 每个彩种只有一个 `collect_url`。
- **改进方案**：为每个彩种配置备用采集 URL 列表，主 URL 失败时自动切换。

---

### 问题 7：台湾彩开奖后延迟回填（5 分钟固定延迟）

- **故障分类**：业务流程设计
- **影响程度**：★★★ 中等
- **影响彩种**：台湾彩
- **故障现象**：台湾彩精确开奖后，`_schedule_backfill_after_draw()` 固定延迟 5 分钟才执行历史回填（预测结果写入）。之前用户看不到带开奖结果的预测资料。
- **根因**：[scheduler.py:959](backend/src/crawler/scheduler.py#L959) `history_backfill_delay_after_draw` 默认 5 分钟。
- **改进方案**：降低默认延迟到 30 秒，确保外部 API 数据落地后再回填。同时添加"外部 API 数据已确认到达"的前置条件检查。

---

### 问题 8：时间精度无保障

- **故障分类**：技术实现缺陷
- **影响程度**：★★ 低
- **影响彩种**：全部
- **故障现象**：`threading.Timer` 的精度受 GIL 和系统负载影响，无法保证秒级精度。`next_time` 的存储格式是毫秒时间戳，但计算链路涉及北京时间字符串 → UTC → 毫秒时间戳的多次转换，引入误差。
- **根因**：[scheduler.py:911-914](backend/src/crawler/scheduler.py#L911) 时区转换链。
- **改进方案**：统一使用 UTC 时间戳存储和计算，仅在展示层转换为北京时间。服务器部署 NTP 时间同步。

---

### 问题 9：无开奖超时应急预案

- **故障分类**：流程缺失
- **影响程度**：★★★★ 严重
- **影响彩种**：全部
- **故障现象**：开奖时间已过但数据迟迟未到，系统缺乏分级响应机制——多少分钟未开盘应告警？多少分钟应人工介入？
- **根因**：无超时 SLA 定义，`alert_draw_staleness` 的阈值不明确。
- **改进方案**：定义三级超时响应策略（见下文"超时应急预案"）。

---

### 问题 10：缺乏开奖审计日志

- **故障分类**：合规缺失
- **影响程度**：★★★ 中等
- **影响彩种**：全部
- **故障现象**：每次开奖的关键时间点（外部 API 请求时间、返回时间、数据入库时间、is_opened 设置时间）没有结构化的审计记录，出问题时难以追溯。
- **改进方案**：新增 `draw_audit_log` 表，记录每个期号的关键操作时间戳。

## 三、超时应急预案

| 超时级别 | 触发条件（开奖时间后） | 动作 |
|---------|---------------------|------|
| **黄色预警** | +30s 数据未入库 | 调度器日志 WARNING；加速轮询到 10s/次 |
| **橙色告警** | +120s 数据未入库 | 自动切换备用采集 URL；发送邮件告警 |
| **红色告警** | +300s 数据未入库 | 邮件 + 推送告警管理员；标记该期为"延迟开盘" |
| **人工兜底** | +600s 数据未入库 | 管理员通过后台"手动录入开奖"功能录入结果 |

## 五、验证标准

1. **开奖数据到达**：开奖时间后 30 秒内 `lottery_draws` 有新记录，号码字段非空。
2. **自动开盘**：数据入库后 5 秒内 `is_opened` 置为 1。
3. **前端可展示**：`is_opened=1` 后前端 API 立即返回该记录。
4. **超时告警**：120 秒未开盘触发邮件告警。
5. **审计完整**：每次开奖在 `draw_audit_log` 有完整时间戳记录。
